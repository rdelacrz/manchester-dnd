# Security, Consistency, and Operations

## Trust boundaries

1. The browser is untrusted.
2. Identity comes only from a valid server-side session resolved from a cookie token digest.
3. Account/campaign/character IDs supplied by a browser identify requested objects; they never establish ownership.
4. The deterministic rules engine, not an LLM or client, decides legal mechanics, dice, modifiers, HP/resource changes, rewards, and BDE cost.
5. MongoDB is an untrusted location for recoverable secrets. It stores password verifiers, random-token digests, ciphertext, and non-secret key IDs—not raw credentials or keys.
6. External text/image providers and object storage are separate trust boundaries. Provider responses are proposals/artifacts until validated and published.

## Gated registration and account authentication

The requirements describe two different concepts that must not be conflated:

- an **admission access token**, generated by the operator and sent to a selected person;
- a **24-hour sign-up session**, created after an access token is successfully presented;
- a normal **account login session**, issued after account creation or credential authentication.

### Admission access-token lifecycle

CLI command (planned):

```bash
cargo run -p manchester-dnd-server --bin mongo-admin -- \
  signup-token generate --role user --expires-in 7d --max-uses 1
```

Flow:

1. CLI generates at least 256 random bits with the OS CSPRNG.
2. CLI prints the raw token once. MongoDB stores only `SHA-256(raw_token)` because the token has high entropy.
3. The user submits the token in an HTTPS form body. Do not place it in a URL, query string, referer, analytics payload, or log.
4. In one transaction, the server checks `state=issued`, expiry, role, and use count; marks it `reserved`; creates one `signup_sessions` document with 24-hour absolute expiry; and writes a minimized audit.
5. The raw sign-up-session token is returned only as a host-only `__Host-...` cookie. A separate CSRF token is required for account creation.
6. The user can retry valid account-creation input within that session. The admission token cannot create a second active sign-up session.
7. Successful account creation atomically marks the access token `consumed`, the sign-up session `completed`, inserts the account, issues a normal account session, and writes the audit/receipt.
8. Expired reservations become unusable. An explicit CLI revoke command sets `state=revoked`; TTL later cleans metadata after the retention grace.

The one-day deadline is enforced by query predicates. TTL cleanup is not immediate and must not be treated as authorization.

### Account creation

Web sign-up may create only `role=user`. Admin creation is CLI-only:

```bash
cargo run -p manchester-dnd-server --bin mongo-admin -- \
  account create-admin --username <name> --email <email>
```

The CLI prompts for the password through a no-echo terminal or accepts it through a documented secret input channel; never a normal command-line flag or environment logged by process inspection. Role change is a separate re-authenticated, audited operator command.

Account creation transaction:

```text
verify signup session and CSRF
verify admission token reservation
normalize username and email
compute email lookup HMAC and randomized ciphertext
hash password outside transaction
start short transaction
  re-check signup session/access token state and expiry
  insert accounts (unique username + email HMAC)
  consume access token
  complete signup session
  insert normal account session
  insert auth audit and exact receipt
commit
set normal session cookie; clear signup cookie
```

Hashing/encryption occurs before the transaction, but every state predicate is rechecked inside it.

### Credential login

Allow either username or email plus password.

- Username lookup: `username_normalized`.
- Email lookup: `HMAC-SHA-256(email_lookup_key, normalized_email)`.
- Return one generic response for unknown account, disabled login, or wrong password.
- Perform a dummy Argon2id verification on lookup miss to reduce timing enumeration.
- Apply HMAC-digested account/network throttle buckets before expensive verification without creating a permanent account-lockout denial of service.
- On success, rehash when stored PHC parameters are below current policy; rotate/revoke sessions after password or privilege changes.

## Password storage

Passwords are **hashed, not encrypted**.

- Use `argon2` and `password-hash` already present in `crates/game-server`.
- Use Argon2id with a unique random salt and one PHC string.
- Benchmark parameters on production-class hardware and cap concurrent hashing work.
- OWASP's published minimum is a floor, not an automatic final setting. Record benchmark evidence for memory, iterations, parallelism, latency, and server concurrency.
- Permit long passphrases and password-manager paste; do not trim or silently Unicode-normalize the password.
- Enforce reasonable minimum/maximum lengths and reject known compromised values if a privacy-preserving check is available.
- Keep an optional pepper outside MongoDB only if rotation and disaster recovery are designed and tested.

## Email encryption and searchable uniqueness

### Clarifying “encrypted DB side”

MongoDB Queryable Encryption and CSFLE encrypt in the trusted application/driver before data crosses the network; MongoDB stores ciphertext. Server-side encryption at rest protects storage media but does not keep plaintext from a compromised database process or privileged database reader.

For this Rust application, use **application/driver-side field encryption so MongoDB never receives plaintext email**.

### Preferred design

For canonical normalized email `E`:

```text
email_lookup_hmac = HMAC-SHA-256(email_lookup_key, E)
email_ciphertext   = XChaCha20-Poly1305(email_data_key, random_nonce, E,
                     AAD = account_id || schema_version || key_id)
```

- Unique-index `email_lookup_hmac`.
- Store ciphertext, random nonce, algorithm, and `key_id`.
- Use separate lookup and encryption keys; do not derive both by string reuse without a reviewed KDF.
- Store keys in a KMS/secret manager or root-readable secret file outside the DB and repository.
- Never log email, HMAC input, ciphertext plaintext, or key material.
- On authorized display, decrypt only the selected account field.
- Key rotation: new writes use the current key ID; a bounded worker decrypts/re-encrypts old records; lookup HMAC rotation requires a dual-index transition and uniqueness check.

A keyed lookup digest prevents an offline database-only attacker from simply hashing a common email dictionary. It still leaks equality patterns and becomes vulnerable if the lookup key is compromised; document this explicitly.

### Why not a unique index on Queryable Encryption ciphertext

MongoDB documents that Queryable Encryption produces different ciphertext for the same plaintext, cannot guarantee plaintext uniqueness through a unique index, and has TTL/unique-index restrictions. It remains an option for other encrypted fields, but email uniqueness still needs a separate keyed lookup value or another authoritative uniqueness service.

## Session and cookie policy

### Raw token handling

- Generate 256 random bits for sign-up, account-session, and CSRF tokens.
- Store only digests in MongoDB.
- Raw bearer token exists only in an `HttpOnly` cookie and transient request memory.
- Raw CSRF token may be rendered/sent separately but is never stored in MongoDB; compare its digest in constant time.
- Do not put tokens in URLs, local storage, logs, SSR hydration payloads, source maps, WASM constants, or analytics.

### Cookie profile

Hosted cookies:

```text
Secure
HttpOnly (bearer cookie)
SameSite=Lax (or stricter where UX permits)
Path=/
no Domain attribute (host-only)
__Host- prefix when applicable
```

Use explicit, separate local-loopback behavior. Hosted startup must fail closed without configured canonical HTTPS origin/host, secure cookie policy, CSRF, and TLS-aware proxy handling.

### Expiry and revocation

- Validate `revoked_at`, `idle_expires_at`, `absolute_expires_at`, account login state, and password/role session version on every authenticated request.
- Advance idle expiry with a bounded write frequency to avoid rewriting on every asset request.
- Rotate at login, privilege change, password change, and sensitive re-authentication.
- Revoke on logout; revoke all older sessions on password reset/change according to policy.
- TTL removes `purge_at` documents eventually but cannot authorize a session.

### DragonflyDB session cache (optional)

When DragonflyDB is enabled, session lookups are cached in DragonflyDB for a short TTL (≤300s) to avoid a MongoDB round-trip on every request. The cache stores only opaque IDs, timestamps, and `password_role_version` — never raw tokens, passwords, or email. On logout, revocation, or password/role change, the corresponding cache key is deleted. The cache TTL is always shorter than the MongoDB idle expiry, and every cache hit still validates expiry and version. If DragonflyDB is unavailable, the middleware falls through to MongoDB. See [07-dragonflydb-integration.md](07-dragonflydb-integration.md).

## Authorization matrix

| Object / operation | User | Campaign owner/GM | Admin |
|---|---|---|---|
| Own account/session | Read/update allowed fields | Same | May perform documented operator actions |
| Player-character library | CRUD own; delete only if no active instance | Own only unless admin | CRUD any with audit |
| Campaign | List member campaigns | Create/update/delete owned; delete only with no open play/encounter | Manage any with audit |
| Campaign membership | Accept/leave self | Invite/remove players, never remove last owner implicitly | Manage any with audit |
| Campaign character instance | Read own/member-public view; mutate only active own turn | Read party-public/GM-safe fields | Manage any with audit |
| Enemy templates | Read permitted public/admin views | Cannot edit global templates initially | Create/update/list/archive |
| Campaign enemy instance | No hidden-state access | Instantiate/generate/manage for owned campaign | Manage any |
| Event templates | Read only if exposed | Cannot edit global templates initially | Create/update/list/archive |
| Campaign event | Member-visible projection | Trigger/resolve within deterministic rules | Manage any |
| Battle/turn command | Current authorized actor only | GM controls flow but cannot forge mechanics | Operator repair only through explicit command |
| BDE | Spend only own balance on accepted custom action | Cannot spend a player's BDE | Grant/refund only through audited policy command |
| Assets/exports/recaps | Authorized owner/member scope | Campaign-owner scope | Explicit audited operator scope |

The apparently conflicting enemy requirement is resolved as follows: **admin edits global enemy templates; a campaign owner may generate/instantiate campaign enemies from approved templates and control campaign-scoped hidden state.** If campaign-owner template authoring is later enabled, it uses owner-scoped templates and review state rather than weakening the global admin boundary.

### Repository query shape

Browser-facing methods take the principal first:

```text
load_owned_character(account_id, character_id)
load_member_campaign(account_id, campaign_id)
load_authorized_character_instance(account_id, campaign_id, instance_id)
commit_member_action(account_id, campaign_id, play_session_id, command)
load_authorized_asset(account_id, asset_id)
```

For opaque IDs, inaccessible and absent objects return the same safe not-found result unless an admin endpoint intentionally distinguishes them.

## Optimistic concurrency

Every mutable aggregate has `revision >= 1`. Use conditional single-document writes:

```text
filter:  _id + ownership/partition + revision: expected_revision + required state
update:  validated field changes + revision: expected_revision + 1 + updated_at
result:  if no match, perform a scoped read to distinguish safe not-found from revision conflict
```

Do not trust a client to supply the next revision. Derived mechanics are recomputed from the loaded authoritative document.

## Multi-document transaction pattern

MongoDB transactions require a replica set or sharded cluster. Development and CI must therefore use a single-node replica set, not standalone `mongod`.

Recommended transaction options:

```text
read concern: snapshot
write concern: majority (journaled according to deployment policy)
read preference: primary
bounded transaction duration and retry count
```

The Rust driver callback may retry transient transaction failures; every callback body must be idempotent and must not perform external side effects.

### Generic authoritative command

```text
1. Authenticate/authorize and load scoped state.
2. Validate command schema and expected revisions.
3. Check receipt; exact fingerprint replay returns stored response.
4. Perform deterministic rules calculation outside the transaction.
5. Start transaction.
6. Re-read/check receipt and all expected revisions/states inside transaction.
7. Apply conditional aggregate updates.
8. Insert append-only turn/audit/ledger rows.
9. Insert exact command receipt last in the same transaction.
10. Commit; on write conflict reload and return/recompute as policy allows.
```

External LLM/image work never runs in steps 5–10.

## Idempotency receipts

Receipt key:

```text
(scope_kind, scope_id, idempotency_key) UNIQUE
```

Receipt content must bind:

```text
actor_account_id
command_kind
canonical request fingerprint
all target IDs
expected revisions/event sequence
result revisions/event IDs
bounded replay response
state: pending|committed|failed where a two-phase workflow requires it
created_at and optional purge_at
```

On key collision:

- identical actor/command/fingerprint/expected state → return prior committed response;
- any mismatch → `IdempotencyConflict`; never treat it as success;
- a pending external-generation receipt resumes/observes the same job rather than enqueueing another;
- destructive receipts live outside the deleted parent and survive long enough to reject replay.

A unique key alone is duplicate suppression, not exact replay.

## Atomic BDE spending

BDE is a campaign-character resource, not an account-wide currency.

Custom-action flow:

1. Show current balance and configured cost.
2. Accept player text only for the active human-controlled slot.
3. Ask the typed interpretation layer to propose an engine command based on current public facts/legal action IDs.
4. Reject invalid, stale, unsafe, or non-applicable output without charge.
5. Begin transaction and re-check actor, turn revision, receipt, character revision, and `runtime.bde.balance >= cost`.
6. Apply mechanics and decrement the materialized balance with compare-and-set.
7. Insert signed `bde_ledger` row with `balance_after`, turn/event links, and exact idempotency key.
8. Insert turn event, minimized audit, and receipt in the same transaction.
9. Exact replay returns the previous result and never spends twice.

AI substitutes cannot spend a human's BDE. Provider timeout/failure, clarification, rejected proposal, transaction conflict, and invalid input cost zero. Refund/cancellation policy is an explicit decision in `06-requirements-and-decisions.md`.

## Randomness and replay

- The server owns CSPRNG/deterministic RNG selection; clients and LLMs never submit outcomes.
- A committed turn stores roll facts: algorithm/version, die sides/count, raw rolls, modifiers, advantage/disadvantage, target/DC, total, and result.
- Store a seed identifier/commitment and protected seed material according to the existing `SeedVault` boundary; do not place reusable secret seeds in ordinary turn documents.
- Retrying the same command returns the recorded result, not a fresh roll.
- Any algorithm/content/rules update is versioned and pinned so old turns can be interpreted.

## Generation queue and publication

### Claim

Use atomic `findOneAndUpdate`:

```text
filter state=queued, available_at<=now, lease absent/expired
sort priority desc, available_at asc, _id asc
set state=running, lease_token_digest, lease_expires_at, worker_id, updated_at
append bounded attempt record
```

### Execute

Provider call occurs outside a MongoDB transaction with a strict deadline, byte/token limits, concurrency limits, and circuit breaker. Raw credentials and private prompts are never persisted.

### Publish

A short transaction verifies job/attempt lease, origin revision/event, moderation result, and idempotency. It inserts presentation/asset metadata, updates the owning entity's selected pointer when applicable, settles governance reservation, writes audit/receipt, and marks the job succeeded.

Object bytes are written to a temporary protected key before publication. Cleanup deletes rejected/orphaned bytes before removing metadata. Character/campaign image routes authorize the owning entity, not merely the asset ID.

## Campaign deletion and retention

### Deletion precondition

Campaign deletion is allowed only when:

- authenticated principal is owner or admin;
- no `play_sessions` document is `waiting|active`;
- no `encounters` document is active;
- expected campaign lifecycle/revision matches;
- a preparation/export policy, if enabled, has been satisfied.

### Explicit deletion manifest

MongoDB has no cascading foreign keys. Keep a tested manifest naming every campaign-owned collection:

```text
campaign_character_instances
campaign_enemy_instances
campaign_events
campaign_invitations
play_sessions
encounters
turn_events
audit_events (delete or minimize by retention policy)
bde_ledger
command_receipts (retain destructive receipts as required)
generation_jobs and reservations
generated_presentations
generated_assets / quarantines
private inspiration campaign consents, vetoes, selections, work
```

Deletion transaction writes a tombstone/receipt and removes bounded Mongo metadata. External asset/export/private-source bytes are deleted or cryptographically erased before the final metadata removal is considered complete. If the operation is too large for one transaction, use a durable deletion job/state machine that first makes the campaign inaccessible, then idempotently cleans each collection, and finally writes completion evidence. Never claim atomic deletion when batch limits make it impossible.

### Account deletion

Define separately:

- campaigns owned by the account: transfer or delete explicitly;
- memberships: remove/minimize;
- reusable characters and campaign instances: delete/retire according to campaign policy;
- sessions/tokens: revoke first;
- email ciphertext and other direct identifiers: cryptographically erase/delete;
- audits: retain only minimized IDs/outcomes where policy permits;
- BDE and turn history: retain campaign facts without recoverable account PII where required.

## MongoDB deployment security

- Enable database authentication/access control; never expose unauthenticated MongoDB.
- Use a least-privilege application role scoped to the one database and required actions.
- Use a separate schema/admin role for collection/index/validator reconciliation.
- Bind local development to loopback; production uses private networking/firewall rules.
- Require TLS in production and validate certificates.
- Redact driver errors/commands that may contain sensitive filters or ciphertext metadata.
- Disable/limit database profiler/query logging for sensitive collections unless redaction is verified.
- Keep MongoDB, KMS, object storage, and LLM credentials in secret management, not `.env` committed to source.
- Set connection/server selection/socket/transaction timeouts and bounded pool sizes.
- Monitor replica-set health, replication lag, transaction aborts, slow queries, TTL backlog, queue leases, document size, and index use.

## Schema bootstrap and change management

Replace SQL migrations with a versioned idempotent schema reconciler:

```text
mongo-admin schema apply
mongo-admin schema verify
mongo-admin indexes verify
```

It must:

1. create missing collections;
2. apply/update validators via `collMod`;
3. create named indexes with exact key/options;
4. detect conflicting/stale indexes rather than silently keeping them;
5. store applied schema bundle version/digest in `system_settings`;
6. refuse startup when required validators/indexes are missing in hosted mode;
7. use explicit upcasters/backfills for future document versions.

Do not rely on application startup to acquire broad schema-admin privileges in production. Development may opt into `MONGO_SCHEMA_APPLY_ON_START=true`; hosted mode defaults to verify-only.

## Backup, restore, and disaster recovery

Back up together:

- MongoDB replica-set-consistent data;
- object/file storage containing images, presentations, exports, and protected private material;
- encryption/KMS configuration and recoverable key backups;
- content packs, GM prompt/policy versions, and RNG vault needed to interpret records;
- schema/index/validator manifest and application release identifier.

Restore test must prove:

1. MongoDB and object metadata/digests agree;
2. encrypted email can be decrypted with restored keys;
3. login sessions are intentionally revoked or restored according to documented policy;
4. campaign/play/encounter state resumes at the same revisions, turn sequence, active actor, HP/resources, BDE, and pinned rules;
5. exact command replay returns the same result;
6. orphaned/missing external artifacts are detected and fail closed;
7. a two-account isolation sample still passes.

Record only minimized backup/restore status in `system_settings.recovery`; backup keys and credentials never go there.

DragonflyDB is explicitly excluded from backup. All cached state is reconstructable from MongoDB. After restoring MongoDB, the application naturally repopulates DragonflyDB on cache misses. Do not include DragonflyDB in the restore verification gate.
