## Executive recommendation

Build a **greenfield MongoDB persistence layer**, not a table-for-collection translation. MongoDB is the system of record; an optional DragonflyDB layer (Redis-compatible, deployed via Coolify) provides session caching, rate limiting, and real-time pub/sub.

- Retain `crates/game-core/` as the deterministic authority for game rules.
- Replace `PostgresRepository` with a neutral Mongo-backed `Repository` facade.
- Keep account-owned, level-less `player_characters` separate from campaign-bound `campaign_character_instances`.
- Embed only bounded data that is loaded and changed as one aggregate: campaign members, play-session participants, content pins, current turn state, and small policy sets.
- Keep independently mutable or unbounded data in separate collections: character instances, encounters, turn history, receipts, audits, BDE ledger, generated artifacts, jobs, and private-inspiration history.
- Use optimistic `revision` compare-and-set for aggregate writes. Use short MongoDB transactions only when one authoritative command must update several documents.
- Run MongoDB as a replica set in development, test, and production because multi-document transactions are required for authoritative turn/BDE commits.
- Store images and other large generated bytes in protected object/file storage. MongoDB stores metadata, ownership, integrity digests, provenance, and lifecycle only.
- Use strict MongoDB `$jsonSchema` validators **and** Rust domain validation. MongoDB has no foreign keys and cannot enforce every cross-document invariant.
- Do not preserve legacy local-owner IDs, SQLite import, PostgreSQL migrations, or SQL compatibility. Preserve useful behavior—authorization, snapshots, audits, exact replay, recovery—not obsolete storage shapes.

## Deliberate non-goals

- No PostgreSQL-to-MongoDB ETL or dual-write period.
- No compatibility layer for migrations `0001`–`0031`.
- No direct persistence of arbitrary LLM output as authoritative mechanics.
- No unbounded arrays of turns, messages, audits, jobs, invitations, or ledger entries inside campaign documents.
- No raw access token, registration token, login session token, CSRF token, password, encryption key, provider credential, or private source body in MongoDB.
- No claim that hidden UI controls provide authorization; every database operation is scoped by the server-derived principal.

## Key architecture decisions

### 1. Use string IDs at the application boundary

Continue opaque, type-prefixed IDs such as `account:<uuid>`, `campaign:<uuid>`, and `encounter:<uuid>` as MongoDB `_id` values. This avoids leaking collection cardinality, preserves current domain validation, and removes repeated `ObjectId` conversion at browser boundaries.

### 2. Version every durable contract

Every aggregate and append-only record carries `schema_version`. Mutable aggregates also carry `revision`. Content/rules references carry an immutable ID, revision, and digest. Upcasters live in Rust; collection validators permit only versions the deployed server understands.

### 3. Pin rules instead of copying the entire rules catalog

Source-controlled, reviewed content packs remain the canonical source for classes, subclasses, spells, equipment, conditions, and deterministic mechanics. A campaign embeds a sealed `rules_snapshot` containing ruleset/content/prompt/policy IDs, versions, and digests. Character/enemy/event instances embed the exact source snapshot needed to remain stable if an external template changes.

### 4. Separate reusable definitions from runtime instances

- `player_characters`: account-owned identity and reusable build choices; no level, XP, current HP, money, BDE, conditions, or campaign inventory.
- `campaign_character_instances`: one campaign-specific realization with source snapshot, level progression, HP/resources, inventory, money, spells, conditions, BDE balance, and revision.
- `enemy_templates`: admin-managed reusable definitions.
- `campaign_enemy_instances`: campaign-scoped snapshots and mutable runtime state.
- `event_templates`: admin-managed event definitions.
- `campaign_events`: selected/instantiated event snapshots and resolution state.

### 5. Make snapshots explicit

Each snapshot includes:

```text
source_id
source_revision
source_schema_version
source_digest
captured_at
payload
```

The snapshot is immutable. Updating a library character, enemy template, event template, rules pack, or prompt does not mutate an active campaign/encounter. A deliberate rebase must be a separate, audited command and is initially unsupported.

### 6. Keep external work outside transactions

Password hashing, LLM calls, image generation, object-storage writes, and large serialization happen before or after a short transaction. The transaction performs only authoritative compare-and-set checks, state mutation, ledger/audit/event insertions, and idempotency receipt publication.

## MongoDB-specific facts that shape this plan

1. A single-document write is atomic; model data together only when it is bounded and shares a lifecycle/write boundary.
2. A BSON document is limited to 16 MiB. Campaign history, narration, audits, and ledgers therefore cannot grow inside one campaign document.
3. Multi-document transactions are available on replica sets and sharded clusters and cost more than single-document writes. They are reserved for true command boundaries.
4. TTL deletion is asynchronous and can lag expiry. Authentication and authorization must check `expires_at`/state themselves; TTL is cleanup, not enforcement.
5. A partial unique index enforces uniqueness only for documents matching its filter. This is useful for one open play session or one active campaign character per account.
6. Queryable Encryption is client/driver-side: the server receives encrypted values. It does not provide reliable plaintext uniqueness for encrypted fields and does not support TTL/unique indexes on Queryable Encryption collections. The preferred email design is therefore randomized AEAD ciphertext plus a separately keyed unique lookup digest.
7. MongoDB does not provide foreign keys or cascade deletion. The service must use transactions, ownership predicates, deletion manifests/tombstones, and consistency checks.

## Recommended initial boundaries

The target design uses 34 collections, not 70. See [01-target-data-model.md](01-target-data-model.md) for details. The highest-risk boundaries are:

- `accounts` + token/session collections;
- `campaigns` with bounded embedded membership and sealed rules snapshot;
- `campaign_character_instances` as the authoritative per-campaign player state;
- `play_sessions` with bounded participant/turn-control state;
- `encounters` for authoritative battle state;
- `turn_events`, `command_receipts`, `audit_events`, and `bde_ledger` as separate append-only histories;
- generation jobs/assets/presentations separated from core mechanics;
- private-inspiration source/consent/work collections separated from ordinary game data.

## Cutover strategy

Because the app is unlaunched and compatibility is unnecessary:

1. Freeze behavioral tests that are still valuable.
2. Add MongoDB infrastructure and schema bootstrap.
3. Replace persistence one vertical slice at a time behind the neutral `Repository` facade.
4. Rewrite tests against disposable MongoDB databases on a single-node replica set.
5. Remove SQLx, PostgreSQL Compose, SQL migrations, role scripts, and legacy import only after every target slice passes.
6. Start with an empty MongoDB database; seed only admin, content metadata, enemy/event templates, and explicit development fixtures through supported CLIs.
7. Keep hosted mode fail-closed until the two-account, transaction, crypto, backup/restore, and protected-asset release gates pass.
