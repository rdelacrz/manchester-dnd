## Existing literals / policy concepts found

- No existing `action_points.rs`, `custom_action_point_ledger`, or product constants for point balance/cost/refund exist yet.
- Existing wording in the plan consistently uses `custom-action points` and `custom-action balance`; avoid introducing a second term such as inspiration because `PrivateInspiration*` already means consented real-life source material in the codebase.
- Existing campaign model is moving from `owner_key = local-owner` to account membership roles `game_master|player`; exactly one owning game-master membership is required by the plan.
- Planned invitation storage already permits either invitee account/email digest or join-code digest, so the first-release mechanism should choose one explicitly rather than implementing both.
- Existing core `MAX_PARTY_CHARACTERS = 64` should not be reused as product policy. It is intentionally broad validation of persisted `SessionDto.character_ids`, while the new lobby/member flow needs a smaller public gameplay cap.

## Recommended provisional policy constants for Task 18

Use one immutable policy object/ID and import constants from one module rather than scattering literals.

Suggested policy ID:

- `CUSTOM_ACTION_POINT_POLICY_ID = "custom-action-points:private-mvp:v1"`

Recommended values:

| Decision | Recommended value | Rationale |
| --- | --- | --- |
| Point name | singular `custom-action point`; plural `custom-action points`; short UI label `Custom-action points` | Matches `.context/REWRITE_PART_1.md` wording and avoids conflict with existing `PrivateInspiration` terminology. |

## Implementation handoff notes

- Create a central core policy module in Task 18, e.g. `crates/game-core/src/action_points.rs`, with public constants and tests for grant/spend/refund/max-balance invariants.
- Treat `earned` as a schema enum reason but reject it under `custom-action-points:private-mvp:v1` unless/until future policy enables earning.
- Task 18 should atomically commit: mechanical turn, spend ledger row, optional materialized balance update, typed-intent receipt, and turn audit. The existing `typed_intent_command_receipts` path proves where the receipt is today; avoid a second best-effort point transaction.
- UI should show `Custom-action points: N / 3` and `Custom action costs 1 point when accepted`; disable custom input at balance 0 while leaving structured legal actions enabled.
- Invitation and max-party constants belong with Task 13/15 campaign policy if those modules already exist by the time Task 18 starts; otherwise Task 18 can depend on the same policy data rather than duplicating values.

## Human decision status

No immediate human block is required. The above values are safe provisional defaults that satisfy the source contract and can be encoded for tests. If the maintainer wants a product-positioning change later, the two most likely alternatives are:

1. Party size: `4` player slots (recommended) vs `5` player slots (larger table, more UI/concurrency complexity).
2. Invitations: normalized-email only (recommended) vs both email and join code (more convenient, higher anti-abuse/security scope).
